**Source project:** [nnTransform3D](https://github.com/simoninns/nnTransform3D)

**Status:** Planning

---

## Overview

This document describes a phased plan to integrate the `nnNtscTransform3D` neural-network NTSC chroma decoder from the `nnTransform3D` project into decode-orc as a standalone **"Sink (3rd party)"** stage. The stage will provide an NTSC-only NN-based chroma decoder for composite and Y/C sources, modelled on the existing `RawVideoSinkStage` but kept deliberately separate so the approach can be proven independently before any deeper integration with the existing `ChromaSinkStage` family.

---

## Background & Design Decisions

### What the nnTransform3D decoder does

The NN decoder in `nnTransform3D` uses a fully 3D (spatial ├Ч spatial ├Ч temporal) comb-filter approach:

1. Extracts 16├Ч16├Ч4-field tiles from composite video (50 % overlap, Hann-windowed)
2. Runs a 3D complex FFT (FFTW3) on each tile
3. Feeds the spectral magnitudes + Hermitian mirror into an ONNX neural network ([`chroma_net_v2.onnx`](https://github.com/simoninns/nnTransform3D/blob/main/nn_training_model/chroma_net_v2.onnx), input `[1,2,4,16,16]` тЖТ mask output `[1,1,4,16,16]`)
4. Applies the learned gain mask to suppress luma and recover chroma
5. Overlap-adds the IFFT results back into a per-frame chroma estimate
6. Post-processes: IQ demodulation, FIR filtering, optional NR, IQтЖТUV rotation

The process requires a look-behind/look-ahead of ┬▒2 fields (1 frame each side), which exactly matches the temporal window available from decode-orc's pipeline when fields are buffered during trigger execution.

### Key architectural constraints

| Constraint | Decision |
|---|---|
| Stage category | `SinkCategory::THIRD_PARTY` тАФ isolated from `CORE` chroma decoders |
| Output formats | RGB48, YUV444P16 (planar), Y4M тАФ same as `RawVideoSinkStage`; no FFmpeg encoding in Phase 1 |
| Video system | NTSC-only (`VideoSystem::NTSC`); reject PAL/PAL-M inputs at execute-time |
| Source type | Composite and Y/C (like all other chroma sinks) |
| Preview | Not provided in Phase 1 тАФ no `PreviewableStage` inheritance; added in Phase 5+ |
| Live preview | Not provided; ONNX inference is expensive, but per-seek preview with a `PreviewDecoderCache` (caching the static FFTW plans and ONNX session) was added post-Phase 1 |
| ONNX model shipping | Embedded as a plain C byte array generated by CMake at configure time (`file(READ HEX)` тЖТ generated `chroma_net_v2_onnx_data.cpp`). No Qt required; the bytes are compiled directly into `orc-core.a`. |
| NN source code placement | Vendored copy inside `orc/core/stages/nn_ntsc_chroma_sink/decoders/` тАФ keeps decode-orc self-contained and avoids submodule churn while the stage is experimental |
| Thread model | Multi-threaded (FFTW plans + ONNX session created lazily via `std::call_once` in static singletons, matching the upstream design). A `--nn-deterministic` equivalent parameter (`deterministic_output`) forces single-thread mode for reproducible results |
| FFTW3 | Already listed in `vcpkg.json`; needs explicit `find_package` + link in `orc/core/CMakeLists.txt` (currently only present for the existing chroma decoder through `orc_chroma_decoders`) |
| ONNX Runtime | Not yet in decode-orc; must be added to `vcpkg.json` and `flake.nix` |

## Data Model Bridge (the key technical challenge)

`nnTransform3D` operates on `ld-decode`'s types (`SourceField`, `LdDecodeMetaData::VideoParameters`, `ComponentFrame`). decode-orc uses its own `VideoFieldRepresentation` (VFR). A thin **adapter layer** must translate between the two:

| ld-decode type | decode-orc equivalent | Notes |
|---|---|---|
| `SourceField::rawBuffer` (`QVector<quint16>`) | VFR field sample data (`std::vector<uint16_t>`) | Direct copy; same 16-bit unsigned encoding |
| `LdDecodeMetaData::VideoParameters` | `SourceParameters` (from `VideoFieldRepresentation`) | Map: `fieldWidth`, `fieldHeight`, `colourBurstStart/End`, `activeVideoStart/End`, `white/black/sampleValueForIreZero` |
| `LdDecodeMetaData::Field.isFirstField` | VFR field polarity / odd-even flag | Determine from field sequence number parity |
| `LdDecodeMetaData::Field.seqNo` | VFR field index | Map directly |

The adapter will live in `orc/core/stages/nn_ntsc_chroma_sink/` as `vfr_to_source_field_adapter.h/.cpp` and will be tested independently in Phase 4.

> **Note:** There is no `.qrc` file. CMake generates `chroma_net_v2_onnx_data.cpp` and `.h` in the build tree at configure time by reading the `.onnx` binary and writing it as a C `const unsigned char[]` array. This keeps `orc-core` free of any Qt dependency.

**Adaptation notes for vendored files:**

- **`comb.h/cpp`** тАФ remove the `DecoderPool` integration; keep `Comb::Configuration` and `Comb::decodeFrames()` unchanged. Replace Qt threading primitives with `std::thread` / `std::once_flag` to remove Qt-Core threading dependencies (decode-orc already provides Qt6 but cleaner to use C++ stdlib here). Leave the FFTW and ONNX calls untouched.
- **`sourcefield.h`** тАФ replace `LdDecodeMetaData::Field` and `LdDecodeMetaData::VideoParameters` with a minimal self-contained `NnSourceField` struct that the adapter (Phase 3) will populate. This severs the ld-decode-tools dependency entirely.
- **`ntscdecoder.h/cpp`** тАФ remove `DecoderPool` threads; the stage's trigger loop will call `Comb::decodeFrames()` directly.
- **`outputwriter.h/cpp`** тАФ replace Qt I/O (`QFile`, stdout pipe) with `std::ostream`-based output to align with the rest of decode-orc's sinks (which use `BufferedFileWriter`).
- **ONNX model path** тАФ do not use Qt resources. Instead, `orc/core/CMakeLists.txt` uses `file(READ HEX)` to read the vendored `.onnx` binary at configure time and generates `chroma_net_v2_onnx_data.cpp` (a plain C `const unsigned char[]` array) in the build tree. `comb.cpp` includes the generated `chroma_net_v2_onnx_data.h` and passes the array pointer directly to the `Ort::Session` from-memory constructor. No `.qrc` file, no Qt dependency in `orc-core`.

**All adapted files carry the original GPLv3 copyright header plus a "Modified for decode-orc" notice.**

**Validation gate:** The vendored files compile cleanly as a static object group (add temporarily as a separate `add_library` to check; remove before Phase 3). No stage registration yet.

### Phase 4 тАФ VFR Adapter & Trigger Implementation

**Goal:** The trigger fully decodes an NTSC composite or Y/C input to a raw video file using the NN comb filter. The adapter correctly bridges decode-orc's `VideoFieldRepresentation` to the NN decoder's `NnSourceField`.

**Adapter tasks:**

- `VfrToSourceFieldAdapter::convert(const VideoFieldRepresentation&, qint32 fieldIndex)` тЖТ `NnSourceField`
- Map `SourceParameters` тЖТ `NnVideoParameters` (the minimal struct defined in Phase 2's `sourcefield.h`)
- Map VFR field data buffer тЖТ `NnSourceField::rawBuffer`
- Map field index parity to `isFirstField`

**Trigger implementation (`NnNtscChromaSinkStage::trigger`):**

1. Validate that the input is NTSC; return early with a clear error message for other video systems
2. Resolve `threads` (0 тЖТ hardware concurrency; `deterministic_output = true` тЖТ 1)
3. Build `Comb::Configuration` from stage parameters
4. Call `comb.updateConfiguration(videoParameters, configuration)`
5. Iterate over fields in the VFR building a `QVector<NnSourceField>` window, with the correct look-behind/look-ahead (┬▒1 frame = 2 fields), calling `Comb::decodeFrames()` per frame
6. Pass each `ComponentFrame` result to the adapted output writer тЖТ write to the output file
7. Report progress via `progress_callback_` every N frames
8. Respect `cancel_requested_` between frames; close and clean up the output file on cancellation

**Output format handling:**

Implement a thin `NnOutputWriter` (derived from/modelled on `nnTransform3D`'s `OutputWriter`) supporting `rgb`, `yuv`, and `y4m` modes, writing to `std::ofstream` via `BufferedFileWriter<uint16_t>`.

**Validation gate:** Full end-to-end encode-decode round-trip test (manual, not automated):
```bash
# Existing ntsc-real.tbc in nnTransform3D/test/ used as reference input
./build/bin/orc-cli --project <ntsc_project> --trigger nn_ntsc_chroma_sink
# Output visually compared against nnTransform3D's own ld-chroma-decoder output
```

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| ONNX Runtime version not available via Homebrew тЙе 1.23.2 | Medium | High | Pin a specific Homebrew tap commit or build from source in CI |
| FFTW plan creation with `std::call_once` conflicts with existing `orc_chroma_decoders` FFTW usage | Low | Medium | Both use separate static planning instances; no shared global FFTW state between modules |
| CMake C array embedding of the `.onnx` model causes binary/compile-time regression | Low | Low | The v2 model is ~654 KB; the generated `.cpp` is ~5 MB of hex literals; compile time is negligible; track size in PR description |
| OLA tile ordering non-determinism under multi-threading produces subtly different output from nnTransform3D's reference | High | Low | Expected; `deterministic_output=true` provides the single-threaded reproducible path for validation |
| `VfrToSourceFieldAdapter` misaligns active video window (off-by-one in `activeVideoStart`) | Medium | High | Cover with unit tests (Phase 5); validate against nnTransform3D reference output on `ntsc-real.tbc` |
| Flatpak sandboxing prevents ONNX Runtime from accessing hardware acceleration backends | Low | Low | ONNX Runtime used in CPU-only mode (`ORT_ENABLE_ALL` optimisations only); no GPU required |
| Windows ONNX Runtime DLL not included in MSI | Medium | High | Explicitly add to CPack component list in Phase 6; test MSI deployment end-to-end |
